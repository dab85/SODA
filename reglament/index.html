Основные правила разработки

Общие правила разработки

  • Доработки осуществляются в основной конфигурации и расширении.

В расширении дорабатываются модули объектов, менеджеров, форм типовых объектов, Типовые общие модули с использованием аннотации &ChangeAndControl, &Before и &After. Использование аннотации &InsteadOf не допускается. Правила разработки в расширении

Во всех остальных случаях доработки осуществляются в основной конфигурации.

  • Префикс проекта: ns_

  • Синоним нового объекта должен оканчиваться на "(ns)"

  • Все новые объекты метаданных создаются с префиксом проекта.

  • Реквизиты объектов добавляются с префиксом проекта, доработка существующих форм — программная, новые формы добавляются через конструктор форм.

Создание новых форм автоматически не допускается.

В методе PrepareForm допускается только добавление реквизитов/элементов, любая видимость/доступность/условное оформление — в отдельных методах.

Типовые роли не меняются, все необходимые права настраиваются в рамках новых ролей.

  • Оформление доработок осуществляется согласно п. 2.2 текущего регламента.

  • Для программного интерфейса описание метода обязательно.

  • Программный интерфейс с директивой Export предполагает регулярное использование.

  • Задачи в работу только от ТА, если не оговорено иное.

  • В хранилище помещается только предварительно протестированная и проверенная синтаксис-проверкой конфигуратора задача. Не допускается помещение в хранилище доработок с ошибками синтаксиса, за исключением случаев, возникающих при использовании методов расширения.

  • Сообщения, заголовки и иная информация для пользователя должна выводиться с использованием функции NStr() по стандарту Интерфейсные тексты в коде: требования по локализации.

Правила доработки конфигурации

Особенности разработки для PostgreSQL

Так как в качестве СУБД используется PostgreSQL, необходимо учитывать особенности, влияющие на производительность:

1.1.1. Использование конструкции FULL OUTER JOIN

В СУБД PostgreSQL реализована только частичная поддержка FULL OUTER JOIN (ERROR: "FULL JOIN is only supported with mergejoinable join conditions"). Для реализации полной поддержки FULL OUTER JOIN при работе 1С:Предприятия 8 с PostgreSQL подобный запрос трансформируется в другую форму с эквивалентным результатом, однако эффективность использования конструкции FULL OUTER JOIN снижается.

В связи с этим не рекомендуется использовать FULL OUTER JOIN при работе с PostgreSQL. В большинстве случаев без использования этой конструкции можно обойтись, переписав исходный запрос.

1.1.2. При работе с PostgreSQL использование соединения с виртуальной таблицей SliceLast может приводить к существенному снижению производительности. Из-за ошибки оптимизатора может быть выбран неоптимальный план выполнения запроса. В связи с этим, если в запросе используется соединение с виртуальной таблицей SliceLast, рекомендуется вынести обращение к виртуальной таблице в отдельный запрос с сохранением результатов во временной таблице.

Использование подсистемы с именем проекта

Все новые объекты необходимо включать в новую подсистему, по возможности в одну из подчинённых. Если объект не подходит ни для одной из подчинённых — включать в основную. Это относится даже к объектам, которые не отображаются в командном интерфейсе (например, общие модули).

Оформление доработок

  1. Код необходимо писать из расчёта, чтобы он мог поддерживаться другими разработчиками.
  2. Все добавленные процедуры и функции в модули типовых объектов должны начинаться с префикса проекта.
  3. При разработке рекомендуется руководствоваться Системой стандартов и методик разработки конфигураций на платформе 1С:Предприятие 8.
  4. В нетиповых модулях необходимо удалять закомментированные участки кода.

Доработки в типовых модулях необходимо помечать комментариями следующего вида:

Начало изменения:

// ++ Korus (Full Name in English, e.g., Ivanov I.I.) Date Task

Окончание изменения:

// -- Korus (Full Name in English, e.g., Ivanov I.I.) Date Task

  1. Все новые модули необходимо размечать областями согласно стандарту «Структура модуля». Весь код модулей необходимо располагать в соответствующих областях. Не допускается размещение модулей вне областей.
  2. Тексты модулей должны быть написаны на русском языке. Исключение составляют веб-сервисы, имена методов и параметров которых рекомендуется задавать на английском языке (например, метод TestConnection веб-сервиса EnterpriseDataExchange), а также различные идентификаторы сторонних информационных систем, например названия полей «внешних» структур данных, которые программно обрабатываются в коде.
  3. Программные модули не должны иметь закомментированных фрагментов кода, а также фрагментов, которые каким-либо образом связаны с процессом разработки (отладочный код, служебные отметки, например TODO, MRG и т.п.) и с конкретными разработчиками этого кода. Тексты модулей оформляются по принципу «один оператор в одной строке».
  4. Текст модуля должен быть оформлен синтаксическим отступом. Для синтаксического отступа используется табуляция (а не пробелы, чтобы при смене числа знаков в табуляции выравнивание текста сохранялось). Желательно использовать отображение символов пробелов и табуляций, чтобы не порождать паразитных вставок их в код.
  5. При длине строки более 120 символов следует использовать переносы. Строки длиннее 120 символов делать не рекомендуется, за исключением тех случаев, когда перенос невозможен (например, в коде определена длинная строковая константа, которая выводится без переносов в окно сообщений с помощью объекта UserMessage).
  6. Тексты модулей могут содержать комментарии. Комментарии должны быть достаточно понятными, чтобы пояснять работу модуля или комментируемого оператора. Тексты комментариев должны составляться по правилам русского языка, в деловом стиле, быть эмоционально сдержанными и не содержать слов, не относящихся к функциональности программы.
  7. Для сложных и нестандартных процедур и функций рекомендуется оформлять описание в соответствии со стандартом Описание процедур и функций.

Снятие с поддержки

Необходимо стремиться к тому, чтобы сохранять объекты на поддержке без включения возможности изменений. Если всё же нет обходного пути, тогда включать возможность изменения можно ТОЛЬКО с сохранением поддержки и только у тех метаданных, которые требуется дорабатывать (не включая подчинённые и родительские объекты метаданных).

Доработка метаданных

Имена новых объектов, имена новых реквизитов типовых объектов и имена предопределённых элементов, макетов, команд и т.д. должны иметь префикс проекта:

  • латиницей;
  • строчными буквами;
  • без символа подчёркивания.

Причём префикс должен быть только на верхнем уровне добавления. Исключение составляют имена web-сервисов и http-сервисов, которые не должны содержать кириллицу в имени. Для таких объектов префикс не требуется. Именование метаданных следует выполнять строго в соответствии со стандартом Имена объектов метаданных в конфигурациях.

Правила создания общих модулей

При вынесении процедур и функций в общие модули необходимо руководствоваться стандартом Правила создания общих модулей. Категорически запрещается создавать общие модули «общего назначения», код в которых может относиться сразу к нескольким функциональным блокам системы.

  • Неправильно: prefixScheduledTasksServer.UpdateProvisionMethod
  • Правильно: prefixProvisionManagement.UpdateProvisionMethod

Модули менеджеров

Необходимо помнить, что модуль менеджера любого объекта метаданных может компилироваться как на клиенте, так и на сервере. В связи с этим ОБЯЗАТЕЛЬНО необходимо обрамлять код абсолютно всех модулей менеджеров соответствующими директивами:

#If Server Or ThickClientOrdinaryApplication Or ExternalConnection Then // код #EndIf

Доработка форм

Все изменения в части интерфейса типовых форм должны вноситься только программно в расширении. Исключение составляют только те редкие случаи, когда платформа не предоставляет возможность программного внесения изменений. Например, расширить тип реквизита формы. В таких исключительных случаях необходимо внести изменение через конструктор формы, а в тексте модуля формы описать внесённое изменение в комментарии. Все программно добавляемые реквизиты и элементы формы должны начинаться с префикса проекта ns_.

При добавлении обработчиков событий в типовых формах имя процедуры должно обязательно начинаться с ключевого слова Attachable и префикса проекта, например Attachable_ns_OnChange. Для новых форм все доработки всегда вносятся через конструктор формы. Необходимо избегать программного создания реквизитов и элементов для нетиповых форм, если только это явно не требуется выполняемой задачей.

Роли

Типовые роли не дорабатываем, за исключением роли FullAccess — для отключения интерактивного удаления новых объектов. Вместо этого необходимо создавать свои роли. При разработке ролей необходимо придерживаться стандарта Настройка ролей и прав доступа.

Разработка регламентных заданий

При разработке регламентных заданий руководствоваться стандартом Регламентные задания, в частности в обязательном порядке использовать в начале метода конструкцию:

CommonFunctions.AtStartOfScheduledJobExecution(Metadata.ScheduledJobs.JobName);

Обработчики обновления данных

Выполнять обновление данных в информационной базе при переходе на новую версию с помощью внешних обработок запрещено. Необходимо пользоваться функциональностью БСП и разрабатывать собственные обработчики обновления для каждой версии. Рекомендуется придерживаться подхода, который используется в типовых конфигурациях — обработчики обновления размещаются в модулях соответствующих объектов в специальных областях. Если при переходе на новую версию нужно обработать несколько объектов, допускается создать обработчик вида OnUpgradeToVersion непосредственно в модуле ns_UpgradeInformationBase в специальной области. При написании обработчиков обновления необходимо:

  • следовать документации по подсистеме;
  • задействовать все возможности подсистемы, например, по отложенному обновлению данных в несколько потоков и блокировке необработанных данных.

ВНИМАНИЕ! Запрещено повторно использовать одинаковые обработчики для обновления на разные версии, в т.ч. «дописывать» имеющийся обработчик для новой версии. Для каждой версии обработчик должен быть написан отдельно.

ВНИМАНИЕ! Необходимо обращать внимание на объём данных в целевой информационной базе. Например, когда в базе разработки справочник содержит 100 элементов, а в основной базе — 2 млн элементов.

ВНИМАНИЕ! Алгоритм получения данных к обработке должен работать таким образом, чтобы уже обработанные данные не перезаписывались при повторном запуске обработчика.

ВНИМАНИЕ! Там, где это возможно, необходимо отключать бизнес-логику и регистрацию объектов в планах обмена. Руководствоваться инструкцией «Массовые изменения данных».

Использование служебных механизмов и библиотек

При доработке конфигурации и расширения доступны следующие механизмы и библиотеки:

  • Расширенные регламентные задания (использовать для добавления новых регламентных заданий)
  • Отложенные задания (использовать для отложенной обработки произвольной логики, где может потребоваться порционная обработка данных)
  • Флаги включения новых алгоритмов (после ввода в эксплуатацию для безопасного запуска новой логики)

По возможности необходимо всегда задействовать эти инструменты, иные случаи должны индивидуально проговариваться с ТА. Для каждого механизма есть отдельная инструкция, с которой необходимо обязательно ознакомиться перед началом работы.
